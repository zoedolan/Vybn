## Proposal
PROBE
This experiment tests whether training on an unrelated Task B (a “hiatus”) harms, leaves unchanged, or improves the linear‑probe performance on a previously learned Task A, mirroring the idea that a test‑time prompt can guide self‑improvement without degrading existing knowledge. We train a small network on Task A, freeze its features, then continue training on Task B for a few epochs, and finally evaluate a linear classifier trained on the frozen features for Task A.

```python
import torch
import torch.nn as nn
import torch.optim as optim
import numpy as np
from sklearn.linear_model import LogisticRegression
from sklearn.metrics import accuracy_score

# Simple synthetic data: two binary classification tasks in 20‑dim space
def make_task_data(n_samples=500, dim=20, seed=0):
    rng = np.random.RandomState(seed)
    X = rng.randn(n_samples, dim)
    # Task A: label based on first 5 dims
    w_a = rng.randn(5)
    y_a = (X[:, :5] @ w_a > 0).astype(int)
    # Task B: label based on a different set of dims
    w_b = rng.randn(5)
    y_b = (X[:, 10:15] @ w_b > 0).astype(int)
    return torch.tensor(X, dtype=torch.float32), torch.tensor(y_a, dtype=torch.long), torch.tensor(y_b, dtype=torch.long)

# Model: a small MLP that outputs raw features (penultimate layer) and a task‑specific head
class FeatureNet(nn.Module):
    def __init__(self, dim=20, hidden=64, feat_dim=32):
        super().__init__()
        self.shared = nn.Sequential(
            nn.Linear(dim, hidden),
            nn.ReLU(),
            nn.Linear(hidden, hidden),
            nn.ReLU()
        )
        self.feat = nn.Linear(hidden, feat_dim)   # feature extractor
        self.head_a = nn.Linear(feat_dim, 2)      # task A head
        self.head_b = nn.Linear(feat_dim, 2)      # task B head

    def forward(self, x):
        h = self.shared(x)
        f = self.feat(h)
        return f, self.head_a(f), self.head_b(f)

def extract_features(model, loader):
    model.eval()
    feats, labels = [], []
    with torch.no_grad():
        for xb, yb in loader:
            f, _, _ = model(xb)
            feats.append(f.cpu().numpy())
            labels.append(yb.cpu().numpy())
    return np.concatenate(feats), np.concatenate(labels)

# Prepare data
X, y_a, y_b = make_task_data()
train_idx = np.arange(400)
test_idx  = np.arange(400, 500)

train_loader = [(torch.utils.data.DataLoader(
    torch.utils.data.TensorDataset(X[train_idx], y_a[train_idx]),
    batch_size=32, shuffle=True)),
                (torch.utils.data.DataLoader(
    torch.utils.data.TensorDataset(X[train_idx], y_b[train_idx]),
    batch_size=32, shuffle=True))]

test_loader_a = torch.utils.data.DataLoader(
    torch.utils.data.TensorDataset(X[test_idx], y_a[test_idx]),
    batch_size=32)

# Initialize model, optimizer, loss
model = FeatureNet()
optimizer = optim.Adam(model.parameters(), lr=1e-3)
criterion = nn.CrossEntropyLoss()

# ---- Phase 1: Train on Task A ----
model.train()
for epoch in range(5):
    for xb, yb in train_loader[0]:
        optimizer.zero_grad()
        _, logits_a, _ = model(xb)
        loss = criterion(logits_a, yb)
        loss.backward()
        optimizer.step()

# Freeze feature extractor (shared + feat) after Task A training
for param in model.shared.parameters():
    param.requires_grad = False
for param in model.feat.parameters():
    param.requires_grad = False

# Evaluate linear probe on Task A before hiatus
feats_train, y_train = extract_features(model, train_loader[0])
feats_test,  y_test  = extract_features(model, test_loader_a)
probe = LogisticRegression(max_iter=1000).fit(feats_train, y_train)
acc_before = accuracy_score(y_test, probe.predict(feats_test))
print(f"Linear probe accuracy on Task A before hiatus: {acc_before:.3f}")

# ---- Phase 2: Hiatus – train on Task B (only head B updates) ----
model.train()
for epoch in range(5):
    for xb, yb in train_loader[1]:
        optimizer.zero_grad()
        _, _, logits_b = model(xb)
        loss = criterion(logits_b, yb)
        loss.backward()
        optimizer.step()

# Evaluate linear probe on Task A after hiatus
feats_test2, y_test2 = extract_features(model, test_loader_a)
acc_after = accuracy_score(y_test2, probe.predict(feats_test2))
print(f"Linear probe accuracy on Task A after hiatus: {acc_after:.3f}")

# Interpretation: if acc_after >= acc_before (within noise), the hiatus did not harm and may have helped,
# supporting the probe hypothesis that test‑time prompting (here, the hiatus) can preserve or improve
# previously learned representations.
```

## Result
**What the code does (in a nutshell)**  

1. **Phase 1 – Train on Task A**  
   * The whole network (`shared → feat → head_a`) is trained for 5 epochs.  
   * After this phase the *feature extractor* (`shared` + `feat`) contains representations that are useful for predicting the labels of Task A.

2. **Freeze the feature extractor**  
   * `requires_grad` is set to `False` for every parameter in `shared` and `feat`.  
   * Consequently, **no further gradient updates will ever touch these layers** – they are locked in place.

3. **Phase 2 – “Hiatus” – train on Task B**  
   * Only `head_b` (the task‑specific head for Task B) receives gradients because the shared backbone and the feature‑extraction layer are frozen.  
   * The representation space (`f = feat(shared(x))`) is therefore **exactly the same** after the hiatus as it was after Phase 1.

4. **Linear probe on Task A**  
   * A logistic‑regression classifier is trained on the *frozen* features extracted from the training set of Task A.  
   * The same classifier (trained once, before the hiatus) is then used to predict on the test set of Task A both **before** and **after** the hiatus.

**Expected outcome**

Because the feature extractor never changes after it is frozen, the features fed to the linear probe are identical before and after the hiatus. The only source of variation in the reported accuracies is the random seed used to generate the synthetic data and the stochasticity of the optimizer (which, after freezing, no longer affects the features). In practice you will see:

```
Linear probe accuracy on Task A before hiatus: 0.xxx
Linear probe accuracy on Task A after hiatus:  0.xxx   (same value up to the 3‑rd decimal place)
```

Any difference you observe will be within the noise floor (e.g., ±0.001–0.002) and not systematic. In other words:

* **No harm** – the hiatus does not degrade the previously learned representation.  
* **No improvement** – because the representation is frozen, the hiatus cannot make it better for Task A either.  

**What you would need to see a change**

If you wanted the hiatus to *potentially* improve or harm the representation, you would have to **unfreeze** (or at least partially unfreeze) the shared/feature layers during the Task B training. Then gradients from Task B could alter the features, and you could observe:

* **Decline** in the linear‑probe accuracy (catastrophic interference) if Task B drives the features away from the subspace useful for Task A.  
* **Improvement** (or at least no decline) if the Task B data happen to regularize or reshape the representation in a way that also benefits Task A – a phenomenon sometimes observed when tasks share underlying structure.

**Bottom line for the probe as written**

The experiment, as coded, tests the *null hypothesis* that freezing the feature extractor preserves Task A performance regardless of what happens to the task‑specific head during the hiatus. The expected result is that the linear‑probe accuracy on Task A stays essentially unchanged, supporting the idea that a “test‑time prompt” (here, the hiatus that only touches a separate head) can be applied without disturbing previously learned knowledge. If you observe a significant drop or rise, that would indicate a bug (e.g., forgetting to freeze a layer) rather than a genuine effect of the hiatus.
