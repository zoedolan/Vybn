The empirical variance across features is **≈ γ²** (here γ = 1.5 → γ² = 2.25). The tiny discrepancy (‑1.3 e‑4) is due only to the default epsilon = 1e‑5 that LayerNorm adds for numerical stability; it is negligible and disappears if you set `eps=0.0`.

**Why this holds for any input distribution**

LayerNorm (with `elementwise_affine=True`) does, for each feature `i`:

1. Compute the mean μᵢ and variance σᵢ² of the activations across the batch.
2. Normalize:  
   \[
   \hat{x}_{b,i}= \frac{x_{b,i}-\mu_i}{\sqrt{\sigma_i^2+\epsilon}}
   \]
   After this step, the variance of `\hat{x}_{:,i}` across the batch is exactly 1 (population variance) – independent of the original distribution of `x`.
3. Scale and shift:  
   \[
   y_{b,i}= \gamma\,\hat{x}_{b,i} + \beta
   \]
   Adding β does not change variance; multiplying by γ scales the variance by γ².

Hence, **the variance of the output of LayerNorm is γ² for every feature, regardless of the input distribution** (as long as the variance used in the normalization step is the population variance, i.e. `unbiased=False`). If you used the unbiased estimator (`unbiased=True`), the variance would be γ²·(N‑1)/N, which converges to γ² as the batch size N grows.

**What you would need to know to answer the probe if you couldn’t run the code**

- The formula for LayerNorm (mean‑variance normalization, scaling by γ, shifting by β).
- That variance is unaffected by an additive constant (β) and scales quadratically with a multiplicative constant (γ).
- Whether the variance computation uses the population or sample estimator (the script uses `unbiased=False`, i.e. population variance).

With that knowledge you can conclude analytically that the post‑LayerNorm variance equals γ², and the empirical test above merely confirms it.
