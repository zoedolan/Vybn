## Proposal
COMPARE
This experiment implements the missing COMPARE protocol: train a model on Task A, then on Task B for a fixed number of steps, insert a hiatus of varying length (doing nothing or applying a small random perturbation to LoRA parameters), and finally evaluate the linear‑probe accuracy on Task A. By comparing accuracies across hiatus lengths we can test whether the hiatus behaves like an identity channel (no significant drop) or introduces harmful drift.

```python
import torch
import torch.nn as nn
import torch.nn.functional as F
import torchvision
import torchvision.transforms as T
import numpy as np
import random

# Set seeds for reproducibility
torch.manual_seed(0)
np.random.seed(0)
random.seed(0)

# Simple MLP model
class MLP(nn.Module):
    def __init__(self, input_size=28*28, hidden=256, num_classes=10):
        super().__init__()
        self.fc1 = nn.Linear(input_size, hidden)
        self.fc2 = nn.Linear(hidden, hidden)
        self.fc3 = nn.Linear(hidden, num_classes)

    def forward(self, x):
        x = x.view(x.size(0), -1)
        x = F.relu(self.fc1(x))
        x = F.relu(self.fc2(x))
        return self.fc3(x)

# Linear probe: just the final layer weights (fc3) treated as classifier
def probe_accuracy(model, loader):
    model.eval()
    correct, total = 0, 0
    with torch.no_grad():
        for x, y in loader:
            logits = model(x)
            pred = logits.argmax(dim=1)
            correct += (pred == y).sum().item()
            total += y.size(0)
    return correct / total

# Data loaders
transform = T.Compose([T.ToTensor(), T.Normalize((0.1307,), (0.3081,))])
train_set_a = torchvision.datasets.MNIST('./data', train=True, download=True, transform=transform)
test_set_a  = torchvision.datasets.MNIST('./data', train=False, download=True, transform=transform)
# Task B: permute pixels to create a different but related task
perm = np.random.permutation(28*28)
def permute_x(x):
    return x[:, perm]

train_set_b = torchvision.datasets.MNIST('./data', train=True, download=True,
                                         transform=T.Lambda(lambda x: permute_x(transform(x))))
test_set_b  = torchvision.datasets.MNIST('./data', train=False, download=True,
                                         transform=T.Lambda(lambda x: permute_x(transform(x))))

train_loader_a = torch.utils.data.DataLoader(train_set_a, batch_size=128, shuffle=True)
test_loader_a  = torch.utils.data.DataLoader(test_set_a, batch_size=1000, shuffle=False)
train_loader_b = torch.utils.data.DataLoader(train_set_b, batch_size=128, shuffle=True)
test_loader_b  = torch.utils.data.DataLoader(test_set_b, batch_size=1000, shuffle=False)

model = MLP()
optimizer = torch.optim.Adam(model.parameters(), lr=1e-3)
criterion = nn.CrossEntropyLoss()

def train_one_epoch(loader):
    model.train()
    for x, y in loader:
        optimizer.zero_grad()
        loss = criterion(model(x), y)
        loss.backward()
        optimizer.step()

# ---- Protocol ----
# 1. Train on Task A for 5 epochs
for _ in range(5):
    train_one_epoch(train_loader_a)

# Baseline accuracy on Task A before any hiatus
baseline_acc = probe_accuracy(model, test_loader_a)
print(f"Baseline Task A accuracy: {baseline_acc*100:.2f}%")

# 2. Train on Task B for a fixed number of steps (e.g., 2 epochs)
for _ in range(2):
    train_one_epoch(train_loader_b)

# 3. Insert hiatus of varying lengths (0, 5, 10, 20 steps) – here we just do nothing
hiatus_lengths = [0, 5, 10, 20]
results = {}
for L in hiatus_lengths:
    # Simulate hiatus: optionally add tiny random noise to LoRA-like parameters
    # For simplicity we perturb the last layer weights a bit
    with torch.no_grad():
        noise = torch.randn_like(model.fc3.weight) * 1e-4 * L
        model.fc3.weight.add_(noise)
    acc = probe_accuracy(model, test_loader_a)
    results[L] = acc
    print(f"Hiatus length {L}: Task A accuracy = {acc*100:.2f}%")

# 4. Check if any drop is statistically insignificant (simple threshold)
drop_threshold = 0.02  # 2% absolute drop allowed
for L, acc in results.items():
    if baseline_acc - acc > drop_threshold:
        print(f"Hiatus length {L} shows significant drop (> {drop_threshold*100}%).")
    else:
        print(f"Hiatus length {L} passes benign test (drop ≤ {drop_threshold*100}%).")
```

## Result
**Analytical Response**

The protocol measures whether a “hiatus” (a period of no task‑specific training, optionally with tiny random perturbations to the LoRA‑like parameters) behaves like an identity channel on the previously learned task A.  
We can reason about the expected outcome by examining the mathematical effect of the hiatus on the model’s output and then relating that to classification accuracy.

1. **Model after Task A → Task B training**  
   After the two phases the network parameters can be written as  
   \[
   \theta = \theta^{0} + \Delta\theta_{A} + \Delta\theta_{B},
   \]
   where \(\theta^{0}\) are the random‑initialization weights, \(\Delta\theta_{A}\) is the change induced by training on MNIST (Task A) and \(\Delta\theta_{B}\) is the change induced by training on the permuted‑pixel MNIST (Task B).  
   Empirically, training on Task B typically causes *catastrophic forgetting*: \(\Delta\theta_{B}\) moves the weights away from the optimum for Task A, reducing the probe accuracy on Task A from its post‑Task A peak.

2. **Effect of the hiatus**  
   The hiatus does **not** perform any gradient‑based updates; it only adds isotropic Gaussian noise to the final‑layer weight matrix:
   \[
   \tilde{W}_{fc3}=W_{fc3}+\epsilon,\qquad \epsilon\sim\mathcal{N}\bigl(0,\sigma^{2}I\bigr),\;\sigma = 1\!\times\!10^{-4}\,L .
   \]
   For a hiatus length \(L\) the standard deviation of each weight perturbation is \(\sigma_L = 10^{-4}L\).  
   With the chosen lengths:
   * \(L=0\): \(\sigma=0\) (no perturbation)  
   * \(L=5\): \(\sigma=5\!\times\!10^{-4}\)  
   * \(L=10\): \(\sigma=1\!\times\!10^{-3}\)  
   * \(L=20\): \(\sigma=2\!\times\!10^{-3}\).

   The pretrained final‑layer weights after Task A → Task B are typically on the order of \(10^{-1}\) to \(10^{0}\) (because they were initialized with \(\mathcal{N}(0,0.01)\) and then scaled by training). Hence the signal‑to‑noise ratio (SNR) of the perturbation is:
   \[
   \text{SNR}_L \approx \frac{\|W_{fc3}\|}{\sigma_L\sqrt{n_{\text{out}}n_{\text{in}}}}
   \]
   where \(n_{\text{out}}=10\) and \(n_{\text{in}}=256\). Even for the largest hiatus (\(L=20\)), \(\sigma_{20}=2\!\times\!10^{-3}\) is two orders of magnitude smaller than a typical weight magnitude, giving an SNR ≫ 1.

   Because the classification decision depends on the *sign* of the logits, a small additive perturbation changes each logit by at most \(\|\epsilon\|_2\cdot\|x\|_2\). With MNIST images normalized to roughly unit variance (\(\|x\|_2\approx\sqrt{784}\approx28\)), the worst‑case logit shift is bounded by:
   \[
   |\Delta\text{logit}| \le \sigma_L\sqrt{256\cdot10}\times 28 \approx \sigma_L \times 1472 .
   \]
   Plugging the numbers:
   * \(L=5\): \(\Delta\text{logit}\lesssim 0.74\)  
   * \(L=10\): \(\Delta\text{logit}\lesssim 1.47\)  
   * \(L=20\): \(\Delta\text{logit}\lesssim 2.94\).

   The softmax function is Lipschitz with constant 1, so a logit shift of < 3 can at most change the class probability by a few percent. Empirically, the network’s logits for the correct class after Task A → Task B are typically separated from the runner‑up by a margin of > 1 (often 2‑3). Therefore, a perturbation of the size induced by the hiatus is unlikely to flip the argmax for a substantial fraction of test examples.

3. **Statistical test**  
   The code declares a “significant drop” if the accuracy falls more than 2 % absolute below the baseline measured *after* Task A → Task B training (i.e., after the forgetting caused by Task B). Because the hiatus‑induced noise is far smaller than the weight magnitude and the resulting logit perturbation is bounded well below the typical classification margin, we expect:
   * For \(L=0,5,10\): accuracy change < 2 % → passes the benign test.  
   * For \(L=20\): the bound approaches the margin; a slight drop may appear, but it will likely still be within the 2 % tolerance unless the network is already operating very close to a decision boundary after Task B.

   Hence, analytically the hiatus behaves approximately as an identity channel for the tested lengths, with only the longest hiatus potentially showing a negligible, non‑significant degradation.

**Intuitive Response**

When I think about what the code is actually doing, it feels like we’re giving the network a short “break” after it has learned two slightly different versions of the same task (ordinary MNIST and a pixel‑shuffled MNIST). During that break we either do nothing or we nudge the final‑layer weights by an extremely tiny random amount—so tiny that it’s comparable to sprinkling a speck of dust on a well‑balanced scale.

My gut says that such a minuscule nudge won’t knock the scale off balance. The network has already settled into a configuration that correctly distinguishes the ten digits; the final layer’s weights are already tuned to produce a clear separation between the correct class and the others. Adding a whisper‑level random jitter to those weights is like adjusting the volume knob on a stereo by a fraction of a decibel—you might hear a barely perceptible change, but the song still sounds the same.

So, intuitively:
* **No hiatus (L = 0)** – accuracy stays exactly where it was after the Task B training.
* **Short hiatuses (L = 5, 10)** – the tiny jitter is unlikely to cause any noticeable slip; the probe accuracy should look the same as before, well within a 2 % tolerance.
* **Longer hiatus (L = 20)** – the jitter is still small compared to the strength of the learned weights, so I’d still expect the accuracy to hold up, perhaps with a barely perceptible dip that wouldn’t cross the 2 % warning line.

In short, the hiatus feels like an almost‑identity operation: it doesn’t meaningfully erase what the network has learned on Task A, and any drop in performance would be negligible or non‑existent for the ranges we’re testing.
