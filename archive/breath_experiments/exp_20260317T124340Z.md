We will train a simple feed‑forward network on a regression task using two optimizers: (1) standard Adam, and (2) a frustration‑aware Adam that detects high‑frustration steps (defined as a sudden increase in loss relative to the previous step) and temporarily reduces the learning rate by half. After a fixed number of epochs we compare the final validation MSE to see if the frustration‑aware optimizer yields better prediction.

```python
import torch
import torch.nn as nn
import torch.optim as optim
import numpy as np

# Synthetic regression data
torch.manual_seed(0)
N_train, N_val = 1000, 200
X_train = torch.randn(N_train, 10)
y_train = X_train @ torch.randn(10, 1) + 0.1 * torch.randn(N_train, 1)
X_val = torch.randn(N_val, 10)
y_val = X_val @ torch.randn(10, 1) + 0.1 * torch.randn(N_val, 1)

train_dataset = torch.utils.data.TensorDataset(X_train, y_train)
val_dataset   = torch.utils.data.TensorDataset(X_val, y_val)
train_loader = torch.utils.data.DataLoader(train_dataset, batch_size=32, shuffle=True)
val_loader   = torch.utils.data.DataLoader(val_dataset, batch_size=32)

# Simple model
class Net(nn.Module):
    def __init__(self):
        super().__init__()
        self.layers = nn.Sequential(
            nn.Linear(10, 50),
            nn.ReLU(),
            nn.Linear(50, 1)
        )
    def forward(self, x):
        return self.layers(x)

def train_model(optimizer_factory, frustration_aware=False):
    model = Net()
    optimizer = optimizer_factory(model.parameters())
    criterion = nn.MSELoss()
    prev_loss = None
    for epoch in range(20):
        model.train()
        for xb, yb in train_loader:
            optimizer.zero_grad()
            pred = model(xb)
            loss = criterion(pred, yb)
            loss.backward()
            # frustration detection: loss increased compared to previous step
            if frustration_aware and prev_loss is not None and loss.item() > prev_loss:
                # temporarily halve LR for this step
                for g in optimizer.param_groups:
                    g['lr'] = g['lr'] * 0.5
            optimizer.step()
            # restore LR after step (if we changed it)
            if frustration_aware and prev_loss is not None and loss.item() > prev_loss:
                for g in optimizer.param_groups:
                    g['lr'] = g['lr'] * 2.0
            prev_loss = loss.item()
    # validation
    model.eval()
    val_losses = []
    with torch.no_grad():
        for xb, yb in val_loader:
            pred = model(xb)
            val_losses.append(criterion(pred, yb).item())
    return np.mean(val_losses)

# run both optimizers
std_mse = train_model(lambda params: optim.Adam(params, lr=1e-3), frustration_aware=False)
frust_mse = train_model(lambda params: optim.Adam(params, lr=1e-3), frustration_aware=True)

print(f"Standard Adam Val MSE: {std_mse:.5f}")
print(f"Frustration‑aware Adam Val MSE: {frust_mse:.5f}")
```

## Result
**Analytical Response**

The data are generated from a *linear* model with additive Gaussian noise:
\[
y = Xw_{\text{true}} + \varepsilon,\qquad \varepsilon\sim\mathcal N(0,0.1^2 I).
\]
The network `Net` contains a hidden ReLU layer, but because the true relationship is linear and the network is over‑parameterized (10 → 50 → 1), there exists a set of weights that yields zero training error (the ReLU can be inactive for all inputs or act as an identity mapping). With a sufficiently small learning rate, Adam will drive the parameters toward a solution that interpolates the training data, and the validation MSE will converge to the noise variance, i.e. approximately \(0.1^2 = 0.01\).

Adam already performs per‑parameter adaptive scaling of the learning rate based on the first‑ and second‑moment estimates of the gradients. When the loss temporarily rises (a “frustration” step), Adam’s internal state will already have reduced the effective step size for those parameters whose recent gradients are large, because the denominator \(\sqrt{\hat v_t}+\epsilon\) grows. Halving the *global* learning rate in response to a raw loss increase therefore:

1. **Duplicates an effect that Adam is already providing**, potentially over‑correcting.
2. **Introduces additional variance** because the loss is noisy (mini‑batch stochasticity). A single‑step increase in loss does not necessarily indicate a bad direction; it can simply be due to sampling noise. Halving the LR on such steps will slow down progress unnecessarily.
3. **Does not address the root cause** of high‑frequency loss oscillations, which are better handled by Adam’s momentum term or by using a learning‑rate schedule.

Consequently, the frustration‑aware modification is expected to either leave the final validation MSE unchanged (if the extra LR halving happens rarely) or to *slightly worsen* it because training proceeds a bit more slowly and may stop before fully exploiting the model’s capacity before the fixed 20‑epoch budget is exhausted. Analytically we therefore predict:

\[
\text{Val MSE}_{\text{standard Adam}} \;\lesssim\; \text{Val MSE}_{\text{frustration‑aware Adam}}.
\]

**Intuitive Response**

Think of Adam as a smart driver that already eases off the gas when the car starts to jerk (the loss goes up) by looking at how steep the road is and how fast the speed is changing. The frustration‑aware tweak is like slamming the brake pedal every time the speedometer shows a tiny bump, even if that bump is just a random wobble caused by a gust of wind (mini‑batch noise). Most of the time the car is already handling the jerk fine, so the extra braking just makes the trip a bit slower. If you only have a short amount of time to reach your destination (20 epochs), arriving a little slower means you might not get as close to the ideal spot, giving a slightly higher error on the test set.

So, gut‑feeling says the plain Adam will probably achieve a marginally lower validation MSE than the frustration‑aware version, though the difference may be small enough that you’d need many runs to notice it.
