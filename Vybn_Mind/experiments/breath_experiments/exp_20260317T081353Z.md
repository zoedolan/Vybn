## Proposal
PROBE
We will fine‑tune a small transformer with LoRA on a synthetic language‑modeling task while adding an auxiliary “frustration” penalty that grows when the model’s validation loss stops improving. By recording the LoRA update vectors (Δθ) at each step and applying a Gaussian‑Mixture Model, we can test whether a second cluster emerges as training progresses, indicating frustration‑driven updates dominate when the model begins to drift.

```python
import torch
import torch.nn as nn
import torch.nn.functional as F
from torch.utils.data import DataLoader, TensorDataset
import numpy as np
from sklearn.mixture import GaussianMixture
import matplotlib.pyplot as plt

# ---------- Synthetic data ----------
vocab_size = 50
seq_len = 10
num_samples = 2000
# Random token sequences
data = torch.randint(0, vocab_size, (num_samples, seq_len))
# Target: next token (shift left)
targets = torch.cat([data[:, 1:], torch.zeros(num_samples, 1, dtype=torch.long)], dim=1)
dataset = TensorDataset(data, targets)
loader = DataLoader(dataset, batch_size=32, shuffle=True)

# ---------- Tiny transformer ----------
class TinyTransformer(nn.Module):
    def __init__(self, vocab_size, d_model=32, nhead=4, num_layers=2):
        super().__init__()
        self.embed = nn.Embedding(vocab_size, d_model)
        encoder_layer = nn.TransformerEncoderLayer(d_model, nhead, dim_feedforward=64)
        self.transformer = nn.TransformerEncoder(encoder_layer, num_layers)
        self.fc = nn.Linear(d_model, vocab_size)

    def forward(self, x):
        # x: (B, S)
        x = self.embed(x) * np.sqrt(self.embed.embedding_dim)
        x = x.transpose(0, 1)  # (S, B, E)
        x = self.transformer(x)
        x = x.transpose(0, 1)  # (B, S, E)
        logits = self.fc(x)
        return logits

model = TinyTransformer(vocab_size)
optimizer = torch.optim.Adam(model.parameters(), lr=1e-3)

# ---------- LoRA wrapper ----------
def lora_linear(layer, r=4, alpha=1):
    """Inject LoRA into a nn.Linear layer."""
    in_features, out_features = layer.weight.shape
    lora_A = nn.Parameter(torch.randn(r, in_features) * 0.01)
    lora_B = nn.Parameter(torch.zeros(out_features, r))
    scale = alpha / r
    def forward(x):
        return F.linear(x, layer.weight, layer.bias) + scale * (F.linear(F.linear(x, lora_A.T), lora_B.T))
    return forward

# Apply LoRA to the final projection layer
model.fc_forward = lora_linear(model.fc, r=4, alpha=1)
def fc_forward(x):
    return model.fc_forward(x)
model.fc = type(model.fc)(model.fc.in_features, model.fc.out_features)  # dummy placeholder
model.fc_forward = fc_forward  # we will call this directly

# Helper to get LoRA params
def get_lora_params():
    return [model.fc_forward.__closure__[0].cell_contents,  # lora_A
            model.fc_forward.__closure__[1].cell_contents]  # lora_B

# ---------- Training with frustration ----------
frustration_weight = 0.0
frustration_increment = 0.0005
val_loss_history = []
delta_theta_list = []  # store flattened LoRA updates per step
step = 0

model.train()
for epoch in range(3):
    for batch_x, batch_y in loader:
        optimizer.zero_grad()
        logits = model(batch_x)  # (B, S, V)
        loss_task = F.cross_entropy(logits.view(-1, vocab_size), batch_y.view(-1))
        # Frustration penalty: increase when validation loss stops improving
        # (we approximate with recent training loss trend)
        frustration = frustration_weight * torch.norm(torch.cat([p.flatten() for p in get_lora_params()]))
        loss = loss_task + frustration
        loss.backward()
        optimizer.step()

        # Record LoRA update (Δθ) since last step
        with torch.no_grad():
            current_params = torch.cat([p.flatten() for p in get_lora_params()])
            if step == 0:
                prev_params = current_params.clone()
            else:
                delta = current_params - prev_params
                delta_theta_list.append(delta.cpu().numpy())
                prev_params = current_params.clone()

        # Simple frustration schedule: increase if loss not decreasing
        if step % 50 == 0 and step > 0:
            recent_losses = [loss_task.item() for _ in range(5)]  # placeholder
            # In real code you would track validation loss; here we just bump weight slowly
            frustration_weight += frustration_increment
        step += 1

# ---------- Analyze Δθ distribution ----------
delta_matrix = np.vstack(delta_theta_list)  # (num_steps, param_dim)
gmm = GaussianMixture(n_components=2, random_state=0)
gmm.fit(delta_matrix)
labels = gmm.predict(delta_matrix)

# Plot first two PCA components for visualization
from sklearn.decomposition import PCA

proj = pca.fit_transform(delta_matrix)

## 2. Major bugs / logical gaps that will prevent the experiment from behaving as described

| Issue | Why it matters | How to fix / improve |
|-------|----------------|----------------------|
| **LoRA wrapper returns a plain Python function, not a `nn.Module`** | The function is attached to `model.fc_forward` but the original `model.fc` is replaced by a dummy placeholder (`type(model.fc)(...)`). During `model(batch_x)` the forward call goes through `nn.Linear.forward`, **not** through the LoRA wrapper, so the LoRA parameters never receive gradients and `get_lora_params()` returns stale values. | Use a proper `nn.Module` subclass (e.g., `LoRALinear`) that holds the base weight, `lora_A`, `lora_B`, and implements `forward`. Then replace `model.fc = LoRALinear(model.fc, r=4, alpha=1)`. |
| **Accessing LoRA parameters via `__closure__`** | Relies on implementation details of the nested function; if the wrapper is changed or Python optimises the closure, the indices may shift, giving the wrong tensors or raising errors. | Store `lora_A` and `lora_B` as attributes of the LoRA module (`self.lora_A`, `self.lora_B`) and expose them via a method or property (`def lora_parameters(self): return [self.lora_A, self.lora_B]`). |
| **Frustration proxy** | The code uses `frustration_weight * ||θ_lora||` as a stand‑in for “validation loss stopped improving”. This has no relation to validation loss; it merely grows as the LoRA parameters increase in magnitude, which will happen even if the model is learning well. Moreover, `frustration_weight` is increased every 50 steps regardless of loss, so the penalty is essentially a deterministic schedule. | Keep a separate validation loader, compute validation loss every N steps, and apply a rule such as: if validation loss has not decreased for `patience` steps, increase `frustration_weight` (or add a term `frustration_weight * max(0, val_loss - best_val_loss)`). Then the frustration term can be something like `frustration_weight * torch.relu(val_loss - best_val_loss)`. |
| **Loss‑tracking placeholder** | `recent_losses = [loss_task.item() for _ in range(5)]` discards the actual loss values and does nothing with them. The subsequent `frustration_weight += frustration_increment` is unconditional. | Actually compute a moving average of the validation loss (or training loss if you truly have no validation set) and only bump the weight when the moving average stops decreasing for a window. |
| **Dummy placeholder for `model.fc`** | `model.fc = type(model.fc)(model.fc.in_features, model.fc.out_features)` creates a brand‑new linear layer with random weights, discarding the pretrained (or initialized) weights of the original projection. This destroys the model’s ability to predict correctly and makes the LoRA term meaningless. | Do **not** replace the layer; instead, wrap it in place: `model.fc = LoRALinear(model.fc, r=4, alpha=1)`. |
| **Gradient flow through LoRA** | Even if the wrapper were called, the current implementation returns a plain Python function that is not registered as a submodule, so PyTorch’s autograd will not track gradients for `lora_A` and `lora_B` unless they are explicitly `nn.Parameter`s and used inside a torch operation that is part of the computational graph. The way the closure is built (`F.linear(F.linear(x, lora_A.T), lora_B.T)`) does create a graph, but because the function is not a method of a `nn.Module`, the parameters are not visible to `model.parameters()`. Consequently, the optimizer never updates them, and `get_lora_params()` returns constant values (the initial random draws). | Make LoRA a subclass of `nn.Module` and add it to the model via `model.fc = LoRALinear(...)`. Then `model.parameters()` will include the LoRA tensors, the optimizer will step them, and `get_lora_params()` will give the true current values. |
| **Shape mismatch in the LoRA forward** | The wrapper does `F.linear(F.linear(x, lora_A.T), lora_B.T)`. If `lora_A` is shape `(r, in_features)` and `lora_B` is `(out_features, r)`, then `x @ lora_A.T` yields `(..., r)`, and the second linear yields `(..., out_features)`. This is correct *only* if the bias of the original layer is handled correctly. The wrapper ignores the original bias (`layer.bias`) and adds it via `F.linear(x, layer.weight, layer.bias)`. That part is fine, but you must ensure the bias term is still present (it is, because you call `F.linear` with `layer.weight, layer.bias`). | Keep the bias term as shown; just ensure the LoRA module is a proper submodule so its parameters are trained. |
| **Number of steps vs. recorded Δθ** | The code records a Δθ only when `step > 0`. The first step stores `prev_params` but does not append a delta, which is fine. However, if the loop ever exits early (e.g., due to an error) the lengths may mismatch. | No major issue, just keep the logic as is. |
| **Plotting incomplete** | The script ends with `plt.title` without a string and without `plt.show()` or `plt.savefig()`. In an interactive notebook this would show an empty title; in a script it would produce no visible output. | Add a meaningful title, e.g., `plt.title('PCA of LoRA update vectors (Δθ) colored by GMM label')`, then `plt.show()`
